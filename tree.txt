TTT-DISCOVER SEARCH TREE (per training step)
=============================================

ARCHIVE = pool of all states found so far (each state = one program + its reward)

                          [ARCHIVE]
                              |
            PUCT picks B parents from the archive
            (high reward + low visit count = picked)
                              |
        +---------+-----------+-----------+---------+
        |         |           |           |         |
     parent0   parent1     parent2     ...      parent(B-1)     <- "groups_per_step"
        |         |           |           |         |
   build prompt from each parent (its code = "previous best")
        |         |           |           |         |
   generate G rollouts each (parallel across GPUs)             <- "group_size"
        |
   +----+----+----+ ... (G children)
   |    |    |    |
  c0   c1   c2  c(G-1)
   |    |    |    |
  run in sandbox -> reward (higher = better)
   |    |    |    |
  valid children pushed back into ARCHIVE
   |    |    |    |
  keep top-K children per parent (rest pruned)


THEN: one gradient step on the LoRA using all rollouts
      update PUCT stats:  n(parent) += G,   T += B*G
      next step starts from the (now bigger) ARCHIVE


HOW A LINEAGE GROWS OVER STEPS
==============================

step 0:   seed
            |
step 1:   seed --> A (reward 1.8)        <- A beat the seed, gets explored next
                    |
step 2:            A --> B (reward 2.1)   <- B is now best in this branch
                          |
step 3:                  B --> C (2.0)    (worse, branch dampened)
                          B --> D (2.3)   <- D best, explored next
                                |
step 4:                        D --> E (2.4)
                                D --> F (2.2)

ARCHIVE after step 4:  {seed, A, B, C, D, E, F}
best so far: E (2.4)


WHY IT'S A TREE, NOT A LINE
===========================
- many parents expanded per step (B branches at once)
- each parent spawns G children (G leaves)
- PUCT balances:  exploit good branches  vs  explore unvisited ones
- lineage blocking stops one step from picking a parent AND its
  own ancestor/descendant in the same batch (keeps branches diverse)
